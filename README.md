# FastAPI Universal Pattern 2026

Универсальный шаблон для создания масштабируемых FastAPI приложений с использованием современных практик и модульной архитектуры.

---

## 🚀 Быстрый старт (Запуск через Docker)

Проект полностью готов к запуску в изолированном окружении Docker. 

### 1. Развертывание (Первый запуск)
Эта команда автоматически создаст все необходимые конфигурационные файлы (включая `.env.development` и `.env.production`) и запустит проект вместе с миграциями баз данных.

**Для Windows / Linux:**
```bash
python scripts/deploy.py
```

**Для Mac OS:**
```bash
python3 scripts/deploy.py
```

**Что происходит при запуске `deploy.py`:**
1. **Создаются файлы окружения:** `.env.development`, `.env.production` и базовый `.env`.
2. **Запускается Docker Compose:** Собирается образ, устанавливаются зависимости через `uv`.
3. **Применяются миграции:** Скрипт `docker-entrypoint.sh` внутри контейнера дожидается готовности БД и автоматически накатывает `alembic upgrade head`.
4. Запускается сервер FastAPI.

*Для остановки всех сервисов:*
```bash
docker compose down
```

## ⚙️ Управление окружениями (.env)

Проект поддерживает разделение настроек. При первом развертывании (`python scripts/deploy.py`) создаются файлы:

1.  **.env.development**: Для локальной разработки.
2.  **.env.production**: Для серверного окружения (требует ручной правки паролей).

### Запуск с выбором окружения
Вы можете явно указать окружение при запуске через `run.py`:

```bash
# По умолчанию используется .env.development
python run.py --env development

# Запуск в режиме продакшена (.env.production)
python run.py --env production
```

*Если вы не указываете `--env`, приложение по умолчанию запустится в режиме `development`.*

## 🏗️ Архитектура

Проект построен на **Модульной (Feature-based) архитектуре**.

```text
src/
├── modules/
│   └── users/              # Весь функционал модуля "Пользователи"
├── core/                   # Общие настройки, базовые классы (BaseRepository)
├── db/                     # Подключения к БД (Postgres, ClickHouse)
└── api/v1/                 # Сборка всех роутеров в единый API
```

## 🛠️ Команды по отдельности

Все команды ниже кроссплатформенны (используйте `python` из корня проекта):

### Запуск API (ручной)
```bash
python run.py
```

### Визуализация (System Map)
```bash
# Генерация данных (после изменения моделей)
python scripts/generate_schema.py

# Запуск сервера просмотра
python scripts/serve_schema.py
```

### Работа с зависимостями (через uv)
Если вы все же хотите разрабатывать локально, рекомендуем использовать `uv`.

```bash
# Инициализация и установка всех зависимостей проекта (создает .venv)
uv sync

# Добавление библиотеки
uv add <package_name>

# Добавление библиотеки для разработки
uv add --dev <package_name>

# Запуск команд через uv (в виртуальном окружении)
uv run pytest
```

### Базы данных (Миграции)
```bash
# Применить миграции
python -m alembic upgrade head

# Создать новую миграцию (авто)
python -m alembic revision --autogenerate -m "Add description"
```

## � Качество кода (Pre-commit)

В проекте настроены автоматические проверки качества кода перед каждым коммитом. Они описаны в файле `.pre-commit-config.yaml` и включают в себя:
- **Black & Isort**: Автоматическое форматирование и сортировка импортов.
- **Ruff**: Быстрый линтинг и исправление типичных ошибок Python.
- **Check-migrations**: Проверка наличия созданных миграций при изменении моделей.

### Автоматический запуск
При выполнении команды `git commit` хуки запускаются автоматически. Если какой-либо инструмент (например, `black` или `ruff`) находит ошибки форматирования, он **автоматически исправит их**, но прервет процесс коммита.

**Пример вывода при ошибке:**
```text
trim trailing whitespace.................................................Failed
- hook id: trailing-whitespace
- files were modified by this hook

black....................................................................Failed
- hook id: black
- files were modified by this hook
reformatted scripts\generate_schema.py
```

**Что делать, если коммит прерван?**
Так как хуки уже внесли правки в файлы, вам достаточно просто добавить эти изменения в индекс и повторить коммит:
1. `git add .` (чтобы добавить авто-исправления в коммит)
2. `git commit -m "ваш комментарий"`

### Ручной запуск
Вы можете запустить все проверки вручную для всех файлов проекта:
```bash
# Для Windows
.\venv\Scripts\python.exe -m pre_commit run --all-files

# Для Linux/macOS
./venv/bin/python -m pre_commit run --all-files
```

## �🛡️ Правила разработки

Все правила описаны в директории `.rules/`.
- **Комментарии и Docstrings**: Всегда пишутся на русском языке.
- **Типизация**: Обязательна (проверяется Mypy).
- **Миграции**: Блокируют коммит, если модели изменены, а миграция не создана.
