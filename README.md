# FastAPI ClickHouse Postgres Pattern 2026

Универсальный шаблон для создания масштабируемых FastAPI приложений.

**Особенность архитектуры:** PostgreSQL развернута локально (в Docker), а ClickHouse и MariaDB рассматриваются как **внешние инфраструктурные сервисы**, к которым приложение подключается для получения или изменения данных.

---

## 🚀 Быстрый старт (Запуск через Docker)

Проект настроен на работу с внешними базами данных. Перед запуском убедитесь, что у вас есть доступы к ним.

### 1. Развертывание (Первый запуск)
Скрипт развертывания автоматически подготовит окружение.

**Для Windows:**
```bash
python scripts/deploy.py
```

**Для macOS / Linux:**
```bash
python3 scripts/deploy.py
```

**Что происходит при запуске `deploy.py`:**
1. **Создаются файлы окружения:** `.env.development` и `.env.production`.
2. **Запускается Docker Compose:** Собирается образ `fastapi_app` и локальная БД `postgres_db`.
3. **Автоматическая инициализация:** Скрипт `docker-entrypoint.sh` внутри контейнера:
    - Дожидается готовности PostgreSQL.
    - Автоматически применяет миграции (`alembic upgrade head`).
    - Генерирует данные для схемы визуализации.
4. **Проверка соединений:** При старте FastAPI проверяет доступность локальной PostgreSQL и внешних MariaDB и ClickHouse.

### 2. Настройка доступов
После генерации `.env.development` откройте его и подставьте **реальные credentials** для подключения к вашим внешним базам:
- `MARIADB_HOST`, `MARIADB_USER`, `MARIADB_PASSWORD`
- `CLICKHOUSE_HOST`, `CLICKHOUSE_USER`, `CLICKHOUSE_PASSWORD`

*Для остановки всех сервисов:*
```bash
docker compose down
```

## 🏗️ Архитектура

Проект построен на **Domain-Driven Design (DDD)** — разделении по бизнес-логике (доменам), а не по технологиям.

```text
src/
├── modules/
│   ├── auth/               # Аутентификация и авторизация (Login)
│   ├── users/              # Управление пользователями и профилями (Postgres)
│   ├── reports/            # Отчеты и связанные фильтры (Postgres)
│   ├── dashboard/
│   │   └── sheets/         # Сервис для работы с таблицами и данными
│   └── storage/            # Хранилище файлов и изображений (S3/Local)
├── core/                   # Общие настройки (Pydantic Settings)
├── db/                     # Клиенты и сессии для каждой БД
└── api/v1/                 # Единая точка сборки роутеров
```

## 🐳 Работа с Docker

В Docker-окружении (`docker-compose.yml`) развернуты:
1.  **app**: FastAPI приложение (доступно по порту **8001**).
2.  **db**: PostgreSQL (основное хранилище приложения).

Внешние базы (ClickHouse/MariaDB) должны быть доступны по сети для контейнера `app`.

## 🛠️ Команды управления

Рекомендуется использовать `uv` для локальной разработки. Внутри Docker `uv` используется автоматически для сверхбыстрой установки зависимостей.

### Визуализация (System Map)
В проекте реализована продвинутая визуализация с тремя режимами просмотра:
1.  **Архитектура**: Общая схема взаимодействия (App <-> DBs).
2.  **PostgreSQL**: Детальная ER-диаграмма локальной базы.
3.  **MariaDB**: Схема данных внешней MariaDB.

```bash
# Генерация данных для схемы (после изменения моделей)
python scripts/generate_schema.py

# Запуск сервера просмотра (доступен на порту 8081 в Docker)
python scripts/serve_schema.py
```

## � Качество кода (Pre-commit)

В проекте настроены автоматические проверки качества кода перед каждым коммитом. Они описаны в файле `.pre-commit-config.yaml` и включают в себя:
- **Black & Isort**: Автоматическое форматирование и сортировка импортов.
- **Ruff**: Быстрый линтинг и исправление типичных ошибок Python.
- **Check-migrations**: Проверка наличия созданных миграций при изменении моделей.

### Автоматический запуск
При выполнении команды `git commit` хуки запускаются автоматически. Если какой-либо инструмент (например, `black` или `ruff`) находит ошибки форматирования, он **автоматически исправит их**, но прервет процесс коммита.

**Пример вывода при ошибке:**
```text
trim trailing whitespace.................................................Failed
- hook id: trailing-whitespace
- files were modified by this hook

black....................................................................Failed
- hook id: black
- files were modified by this hook
reformatted scripts\generate_schema.py
```

### Почему хуки возвращают Failed?
Поведение, когда вы видите `Failed` — это абсолютно нормальная и **задуманная механика работы `pre-commit`**.
Слово `Failed` в логах не означает, что код сломан. Это означает: *"Я нашел несоответствия стандартам, я их автоматически исправил, но я прерываю коммит, чтобы ты мог посмотреть, что именно я изменил"*.

**Как это работает:**
1. Вы пишете код и делаете `git commit`.
2. `pre-commit` перехватывает коммит и запускает хуки (`black`, `isort`, `ruff`).
3. Если код не идеален (например, лишний пробел или неправильный порядок импортов), хуки **автоматически правят файлы** на диске.
4. Поскольку файлы изменились, хуки возвращают статус `Failed` (ошибка проверки) и **прерывают коммит**. Это сделано специально для безопасности, чтобы Git не закоммитил изменения, которые вы не видели.
5. Вам нужно просто сделать `git add .` (чтобы добавить автоисправления в индекс) и заново запустить `git commit`.

**Что делать, если коммит прерван?**
Так как хуки уже внесли правки в файлы, вам достаточно просто добавить эти изменения в индекс и повторить коммит:
1. `git add .` (чтобы добавить авто-исправления в коммит)
2. `git commit -m "ваш комментарий"`

### Ручной запуск
Вы можете запустить все проверки вручную для всех файлов проекта через стандартный Python или `uv`:

```bash
# Через стандартный Python (рекомендуется)
python -m pre_commit run --all-files

# Через uv (если установлен)
uv run pre_commit run --all-files
```

## �🛡️ Правила разработки

Все правила описаны в директории `.rules/`.
- **Комментарии и Docstrings**: Всегда пишутся на русском языке.
- **Типизация**: Обязательна (проверяется Mypy).
- **Миграции**: Блокируют коммит, если модели изменены, а миграция не создана.
